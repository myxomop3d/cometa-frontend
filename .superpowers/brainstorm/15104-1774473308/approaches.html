Architecture Approaches

How to structure the new filter system for BoxDto and reuse across pages

A

Extend SimpleTable with new filter types

Add filter type definitions to SimpleTable's filterFields prop. Each field declares its type (text, number-range, checkbox, select, date-range, relation-dropdown, relation-modal, relation-both). SimpleTable renders the right input component based on type.

Pros

  • Single component handles everything
  • Consistent with current pattern
  • Filter state stays URL-synced via existing useFilters

Cons

  • SimpleTable grows large with many filter types
  • Relation filters need query hooks — tight coupling
  • Hard to test individual filter types
B

Composable filter components + enhanced SimpleTable

Create individual filter components (NumberRangeFilter, DateRangeFilter, RelationFilter, etc.) that each manage their own UI but report values through a shared onFilterChange callback. SimpleTable accepts a filterSlot render prop or children instead of building filters internally. The route page composes the filter bar.

Pros

  • Each filter is an independent, testable component
  • SimpleTable stays focused on table rendering
  • Easy to add new filter types without touching SimpleTable
  • Relation filters can use their own query hooks cleanly

Cons

  • More files to create
  • Page component has more wiring code
C

Config-driven filter builder

Define a filter schema object (similar to editConfig) that declaratively describes all filters. A FilterBuilder component reads the schema and renders the right inputs. Relation configs include query functions and display extractors.

Pros

  • Very declarative — adding a page is just writing a config
  • Uniform API for all filter types

Cons

  • Abstracts too early — only 2 pages so far
  • Config objects become complex (query fns, display fns, modal configs)
  • Harder to customize per-page edge cases

Recommendation: Option B — composable filter components. It keeps SimpleTable clean, each filter type is self-contained, and relation filters with their query hooks stay decoupled. The slight extra wiring in the page component is worth the modularity.
