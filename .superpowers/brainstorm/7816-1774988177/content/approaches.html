Architecture Approaches for box-dice page

How to integrate tablecn's data grid with your server-side pattern

A

Thin Adapter Layer

Copy tablecn's data grid components as-is into src/components/data-grid/. Create a thin useServerDataGrid hook that replaces TanStack DB with your existing TanStack Query + URL-synced filters pattern. The hook translates grid state (sorting, filtering, pagination) into your OData API params.

Pros

  • Minimal changes to tablecn components
  • Easier to pull upstream updates
  • Clean separation: grid UI vs data fetching

Cons

  • Unused code paths remain in components
  • Larger bundle — features you don't use still ship
B

Trimmed Fork

Copy only the components you need from tablecn, removing undo/redo, paste, dnd, cell range selection, and TanStack DB references. Adapt the useDataGrid hook to accept server-side state directly. Keep cell variants, column header, filter/sort/view menus.

Pros

  • No dead code — every line serves your use case
  • Smaller bundle and simpler debugging
  • Full control over the implementation

Cons

  • More upfront work to strip and adapt
  • Harder to pull upstream changes
C

Inspired Rebuild

Don't copy tablecn code. Instead, use it as a reference to build your own data grid components from scratch, using the same patterns (TanStack Table + virtual scrolling + cell variants) but designed from the start for your server-side architecture.

Pros

  • Purpose-built for your exact needs
  • No adaptation friction
  • Deepest understanding of every line

Cons

  • Significantly more work
  • Risk of missing edge cases tablecn already handles
  • Cell editing UX is hard to get right from scratch
